# 구글 인앱 결제 및 애드몹 연동 상세 기술서

이 문서는 `purchase.md`의 각 단계를 구현하기 위한 상세 코드 스니펫과 설정 정보를 포함합니다.

---

### 1-1. 구글 애드몹 설정 (광고 환경 구축)

#### ① 애드몹 로그인 및 앱 등록
1. 구글 애드몹 콘솔에 구글 계정으로 로그인합니다.
2. 왼쪽 메뉴에서 **앱 > 앱 추가**를 클릭합니다.
3. 플랫폼은 **Android**를 선택합니다.
4. '앱이 지원되는 앱 스토어에 등록되어 있나요?' 질문에 **아니요**를 선택합니다. (출시 전 단계 기준)
5. 앱 이름을 입력하고 앱 추가를 완료합니다.
6. 완료 화면에서 `ca-app-pub-XXXXXXXXXXXXXXXX~XXXXXXXXXX` 형태의 **앱 ID(App ID)**가 발급됩니다. 이 값을 따로 메모해 둡니다.

#### ② 광고 단위 생성 (배너 및 전면 광고)
광고 제거 상품을 테스트하기 위해 앱에 띄울 광고 단위를 만듭니다. 보통 하단 배너나 화면 전환 시 나오는 전면 광고를 많이 씁니다.
1. 생성한 앱의 메뉴에서 **광고 단위 > 광고 단위 추가**를 클릭합니다.
2. **배너(Banner)**를 선택하고 광고 단위 이름을 'Main_Bottom_Banner' 등으로 입력한 뒤 생성합니다.
3. 생성 직후 `ca-app-pub-XXXXXXXXXXXXXXXX/XXXXXXXXXX` 형태의 **광고 단위 ID(Ad Unit ID)**가 발급됩니다.
4. 필요하다면 같은 방식으로 전면(Interstitial) 광고 단위도 추가로 생성합니다.

---

### 1-2. 구글 플레이 콘솔 설정 (결제 환경 구축)

#### ① 개발자 계정 및 앱 생성
1. Google Play 콘솔에 로그인합니다. (최초 이용 시 개발자 등록비 약 $25 필요)
2. 우측 상단의 **앱 만들기** 버튼을 클릭합니다.
3. 앱 이름, 기본 언어(한국어), 앱 또는 게임(앱), 유료 또는 무료(무료)를 선택합니다.
4. 개발자 선언문 및 미국 수출법 동의란에 체크한 뒤 앱 만들기를 완료합니다.

#### ② 인앱 상품 (광고 제거 단품) 생성
광고 제거는 한 번 구매하면 평생 유지되는 비소모성 단품 상품으로 등록해야 합니다.
1. 왼쪽 대시보드 메뉴를 아래로 내려 **수익 창출 > 상품 > 인앱 상품**으로 이동합니다.
2. **상품 만들기** 버튼을 클릭합니다.
3. **제품 ID(Product ID)**를 입력합니다. (예: `remove_ads_premium`)
   - 이 ID는 추후 Kotlin 코드에서 결제를 요청할 때 직접 사용하므로 기록해 두세요.
4. 상품 세부정보에 사용자가 결제창에서 볼 이름과 설명을 적습니다. (예: '광고 제거 패키지', '앱 내 모든 광고를 영구히 제거합니다.')
5. **가격 설정**을 클릭하여 판매 금액(예: 4,900원)을 입력합니다.
6. 우측 하단의 저장을 누른 후, 상단의 **활성화** 버튼을 반드시 눌러야 결제 테스트가 가능해집니다.

### 1-3. 프로젝트 연동 준비 체크리스트
플랫폼 설정이 끝났다면 Kotlin 코딩을 시작하기 전 다음 표를 채워둘 수 있어야 합니다.

| 항목 | 형식 예시 | 실제 내 발급 값 |
| :--- | :--- | :--- |
| 애드몹 앱 ID | `ca-app-pub-1234~5678` | |
| 배너 광고 ID | `ca-app-pub-1234/9999` | |
| 인앱 상품 ID | `remove_ads_premium` | |

## 2. 광고 및 결제 환경 설정 (Project Setup)

### 2-1. 빌드 환경 설정 (app/build.gradle 또는 app/build.gradle.kts)
앱 모듈 단위의 빌드 스크립트 파일에 광고와 결제 기능을 사용할 수 있도록 외부 라이브러리(SDK)를 등록해야 합니다.
1. **프로젝트 트리에서 파일 열기:** `app` 폴더 내에 있는 `build.gradle` (또는 Kotlin DSL의 경우 `build.gradle.kts`) 파일을 엽니다.
2. **의존성(dependencies) 블록 이동:** 파일 하단의 `dependencies { ... }` 내부에 다음 두 가지 라이브러리를 추가합니다.
   - **구글 플레이 결제 코루틴 확장 라이브러리:** 결제 프로세스를 Kotlin의 코루틴 체계 속에서 비동기 처리하기 위해 일반 billing 대신 `billing-ktx` 라이브러리를 사용합니다.
   - **구글 모바일 광고 SDK:** 애드몹 광고를 불러오고 노출하는 데 필요한 `play-services-ads` 라이브러리를 사용합니다.
3. **그레이들 동기화:** 상단에 나타나는 **[Sync Now]** 링크를 클릭하여 수정한 설정을 프로젝트에 반영합니다.

```kotlin
// 예시: build.gradle.kts (Kotlin DSL)
dependencies {
    // 구글 플레이 결제 (Billing)
    implementation("com.android.billingclient:billing-ktx:7.0.0")
    // 구글 모바일 광고 (AdMob)
    implementation("com.google.android.gms:play-services-ads:23.0.0")
}
```

### 2-2. 앱 권한 및 메타데이터 설정 (AndroidManifest.xml)
구글 시스템이 앱의 광고 권한과 결제 모듈, 그리고 앱의 신원을 식별할 수 있도록 매니페스트 파일을 수정해야 합니다.
1. **기본 권한 지정:** `<manifest>` 태그 바로 아래에 앱이 외부와 통신할 수 있는 필수 권한을 선언합니다.
   - **INTERNET 권한:** 구글 광고 서버 및 플레이 스토어 결제 서버와 데이터를 주고받기 위해 반드시 필요합니다.
   - **AD_ID 권한:** 안드로이드 12(API 레벨 31) 이상 기기에서 광고 수신을 위한 광고 ID(Advertising ID)에 접근하기 위해 반드시 필요합니다.
2. **애드몹 앱 ID 메타데이터 등록:** `<application>` 태그 내부에 `<meta-data>` 태그를 추가합니다.
   - 속성 이름(`android:name`)은 반드시 `com.google.android.gms.ads.APPLICATION_ID`로 정확하게 입력해야 합니다.
   - 속성 값(`android:value`)에는 1단계에서 발급받은 '애드몹 앱 ID'(`ca-app-pub-~`)를 그대로 붙여넣습니다. 이 값이 없거나 틀리면 앱 구동 시 즉시 크래시(App Crash)가 발생합니다.

```xml
<manifest ...>
    <uses-permission android:name="android.permission.INTERNET" />
    <uses-permission android:name="com.google.android.gms.permission.AD_ID" />

    <application ...>
        <meta-data
            android:name="com.google.android.gms.ads.APPLICATION_ID"
            android:value="ca-app-pub-XXXXXXXXXXXXXXXX~XXXXXXXXXX"/>
    </application>
</manifest>
```

### 2-3. 개발용 테스트 광고 ID 준비 (정책 위반 방지)
개발 및 테스트 단계에서 본인이 발급받은 실제 광고 단위 ID를 사용하여 광고를 로드하고 클릭하면 구글 정책상 '무효 트래픽'으로 간주되어 애드몹 계정이 영구 정지될 수 있습니다. 출시 전까지는 구글이 공식 제공하는 공용 테스트 ID를 매칭해 두어야 합니다.

- **안드로이드 공용 테스트 배너 광고 ID:** `ca-app-pub-3940256099942544/6300978111`
- **안드로이드 공용 테스트 전면 광고 ID:** `ca-app-pub-3940256099942544/1033173712`

> [!TIP]
> 코드 작성 단계에서는 위의 범용 테스트 ID를 변수에 담아 서빙하고, 앱을 구글 플레이 스토어에 프로덕션 배포하는 최종 빌드 시점에만 1단계에서 발급받은 실제 광고 ID로 교체하도록 설계하는 것이 안전합니다.

## 3. 광고 및 결제 구현 (Implementation)

### 3-1. 광고 시스템 초기화 및 UI 배치
#### ① 광고 SDK 초기화
- **호출 시점:** 앱이 실행되는 최초 진입점인 `Application` 클래스 또는 메인 `Activity` 단계에서 실행해야 합니다.
- **동작 방식:** 모바일 광고 SDK의 초기화 함수를 호출합니다. 이 작업은 비동기로 진행되며, 완료 콜백을 받은 이후에 실제 광고 요청을 보낼 수 있습니다.

#### ② 광고 UI 배치 및 로드
- **레이아웃 구성:** XML 화면 설계 파일 하단에 광고 전용 뷰 레이아웃(`AdView`)을 배치합니다. 크기는 가장 범용적인 '스마트 배너' 형식을 주로 사용합니다.
- **광고 호출:** 뷰 객체를 참조하여 빌더 패턴으로 생성한 광고 요청(`AdRequest`) 객체를 전달합니다. 이때 2단계에서 준비한 테스트 광고 ID를 매핑하여 서버로부터 광고 데이터를 수신합니다.

### 3-2. 인앱 결제 (BillingClient) 핵심 로직 구현
#### ① 결제 클라이언트 초기화 및 연결
- **클라이언트 생성:** 빌더를 통해 `BillingClient` 객체를 생성합니다. 이때 결제 결과(구매 성공/실패)를 실시간으로 감지할 수 있는 청취자(`PurchasesUpdatedListener`)를 반드시 함께 등록해야 합니다.
- **스토어 연결:** 연결 시작 함수를 호출하여 구글 플레이 앱과의 통신을 개시합니다. 연결이 성공(기기 내 구글 계정 정상 연동)해야만 다음 단계를 진행할 수 있습니다.

#### ② 광고 제거 상품 정보 조회
- **세부 정보 요청:** 1단계에서 설정했던 단품 상품 ID(`remove_ads_premium`)와 상품 유형(인앱 상품)을 파라미터로 구성합니다.
- **비동기 쿼리 수행:** 구글 플레이 서버에 해당 상품의 최신 가격, 통화 기호, 활성화 상태 등을 비동기로 조회하여 화면에 표시할 준비를 합니다.

#### ③ 결제 프로세스 시작 (구매 창 띄우기)
- **결제 요청 생성:** 조회된 상품 상세 정보 객체를 기반으로 결제 플로우 파라미터를 작성합니다.
- **구글 결제창 실행:** 화면을 제어하는 액티비티 컨텍스트와 파라미터를 넘겨 결제 실행 함수를 호출합니다. 이때 앱 화면 위로 구글 플레이 표준 결제 바텀시트가 팝업됩니다.

#### ④ 결제 결과 처리 (콜백 청취)
- **결제 상태 검증:** 사용자가 결제를 마치면 초기화 단계에서 등록했던 청취자로 결과가 전달됩니다. Response Code가 성공(`OK`)이고, 구매 데이터가 유효한지 확인합니다.

### 3-3. 즉시 승인 및 구매 내역 복원 (비소모성 처리)
#### ① 구매 즉시 승인 (Acknowledgment)
- **상태 확인 및 승인:** 결제가 완료되면 해당 구매 건의 승인 여부를 체크합니다. 광고 제거는 단 한 번만 구매하는 비소모성 상품이므로, 구매가 성공한 즉시 `acknowledgePurchase` 함수를 호출하여 승인 처리를 완료해야 합니다.
- **영구 소유권 확정:** 이 승인 절차가 누락되면 구글 시스템은 결제에 오류가 생긴 것으로 판단하여 3일 이내에 금액을 강제 환불하고 아이템을 회수합니다.

#### ② 기존 구매 복원 (Query Purchases)
- **소유권 재확증:** 사용자가 앱을 삭제 후 재설치하거나 기기를 변경했을 때를 대비하는 필수 로직입니다. 앱이 실행될 때마다 현재 로그인된 구글 계정의 유효한 인앱 상품 구매 내역이 있는지 조회합니다.
- **과거 내역 확인:** 리스트에 광고 제거 상품 ID가 포함되어 있다면, 사용자가 과거에 이미 결제 완료한 상태이므로 별도의 결제 유도 없이 즉시 광고 제거 상태를 적용합니다.

### 3-4. 데이터 연동 (광고 숨김 제어 로직)
#### ① 상태 변수 관리 (isAdFree)
- **플래그 설정:** 앱 내부(예: ViewModel 등)에 광고 제거 여부를 판별하는 부울(Boolean) 상태 변수를 선언합니다. 기본값은 `false`입니다.
- **로컬 저장소 동기화:** 구매가 성공하거나 복원 로직을 통해 소유권이 확인되면 이 변수를 `true`로 변경하고, 앱을 재시작해도 유지되도록 암호화된 로컬 저장소(`SharedPreferences` 등)에 해당 상태를 영구 저장합니다.

#### ② UI 바인딩 및 광고 차단
- **뷰 제어:** 상태 변수의 값이 `true`로 감지되면 레이아웃에 배치했던 광고 전용 뷰의 가시성(Visibility)을 `GONE` 상태로 변경하여 화면에서 완전히 숨기고 영역을 지웁니다.
- **호출 차단:** 상태가 `true`인 경우, 광고를 새로고침하거나 서버에 요청하는 로직 자체가 실행되지 않도록 조건문 처리하여 불필요한 네트워크 트래픽을 원천 차단합니다.

## 4. 인앱 상품 활성화 및 빌드 업로드 (Release Preparation)

### 4-1. 결제 권한이 포함된 최초 빌드 파일(AAB) 생성
인앱 상품을 등록하기 전에 구글 시스템에 "이 앱은 인앱 결제 기능을 사용하는 앱이다"라고 선언하는 과정입니다.
- **빌드 버전 확인:** `app/build.gradle` 파일에서 `versionCode`와 `versionName`이 초기 값(예: 1, "1.0.0")으로 잘 설정되어 있는지 확인합니다.
- **배포용 번들 생성:** 안드로이드 스튜디오 상단 메뉴에서 **Build > Generate Signed Bundle / APK...**를 선택합니다.
- **Android App Bundle(AAB) 선택:** 플레이 스토어 업로드 표준 포맷인 **Android App Bundle**을 선택하고 다음으로 진행합니다.
- **키스토어(Keystore) 서명:** 앱의 고유 신원을 보증할 보안 키 파일(`.jks`)을 생성하거나 기존 키를 선택합니다. 비밀번호와 별칭(Alias)을 입력합니다.
- **릴릴스(Release) 빌드 수행:** 빌드 타입을 **release**로 선택하고 생성을 완료하면 프로젝트 폴더 내에 `.aab` 확장자를 가진 최종 앱 파일이 생성됩니다.

### 4-2. 구글 플레이 콘솔 내부 테스트 트랙에 업로드
일반 사용자에게 노출되지 않고, 지정한 개발자와 테스터만 안전하게 앱을 다운로드하여 결제 기능을 검증할 수 있는 환경을 만듭니다.
- **내부 테스트 메뉴 진입:** Google Play 콘솔의 해당 앱 대시보드에서 왼쪽 메뉴의 **테스트 > 내부 테스트(Internal testing)**를 클릭합니다.
- **새 버전 출시:** 우측 상단의 **새 버전 출시 만들기(Create new release)** 버튼을 클릭합니다.
- **AAB 파일 업로드:** '앱 번들' 업로드 영역에 앞서 안드로이드 스튜디오에서 생성한 배포용 `.aab` 파일을 드래그 앤 드롭으로 업로드합니다.
- **권한 자동 인식 확인:** 업로드가 완료되면 구글 시스템이 파일 내부를 분석합니다. 이때 2단계에서 설정했던 결제 라이브러리로 인해 `com.android.vending.BILLING` 권한이 앱에 포함되어 있음이 콘솔에 자동으로 기록됩니다.
- **출시 완료:** 버전 이름과 간단한 출시 노트(예: "최초 결제 테스트 버전")를 입력한 후, 하단의 **버전 검토 > 내부 테스트 트랙으로 출시 시작**을 눌러 저장을 완료합니다.

### 4-3. 인앱 상품 생성 및 활성화
앱 파일이 성공적으로 인식되었으므로, 기존에는 비활성화되어 있거나 생성이 불가능했던 인앱 결제 상품을 최종 승인하는 단계입니다.
- **인앱 상품 메뉴 이동:** 왼쪽 메뉴에서 **수익 창출 > 상품 > 인앱 상품(In-app products)**으로 이동합니다. (이제 빌드가 등록되었으므로 정상 접근 및 생성이 완전히 활성화됩니다.)
- **상품 만들기:** 상품 만들기 버튼을 누르고, 기획하신 광고 제거 상품 ID(예: `remove_ads`)를 입력합니다.
- **메타데이터 작성:** 사용자가 구글 결제창에서 보게 될 '상품 이름'(예: 광고 제거 패키지)과 '설정 설명'을 명확히 작성합니다.
- **가격 설정 및 국가 지정:** **가격 책정 수정**을 눌러 현지 통화 기준으로 금액을 지정합니다. 구글이 자동으로 국가별 환율을 계산해 줍니다.
- **최종 활성화:** 우측 하단의 저장을 누른 후, 화면에 나타나는 **활성화(Activate)** 버튼을 반드시 클릭합니다. 상태가 비활성에서 활성으로 변경되어야 비로서 기기에서 결제 호출을 보냈을 때 구글 서버가 상품 정보를 반환합니다.

## 5. 통합 테스트 및 최종 검증 (Verification & Testing)

### 5-1. 광고 노출 테스트 (기본 화면 검증)
실제 프로덕션 배포 전, 개발 중인 앱에서 광고 레이아웃이 깨지지 않고 정상적으로 수신되는지 확인합니다.
- **테스트 광고 확인:** 앱을 실행했을 때 2단계에서 설정한 공용 테스트 ID에 의해 "Test Ad"라는 라벨이 붙은 샘플 배너나 전면 광고가 정상적으로 화면에 나타나는지 확인합니다.
- **에러 코드 체크:** 광고가 나오지 않는다면 안드로이드 스튜디오의 **Logcat** 탭에서 `Ads` 키워드로 검색하여 구글 서버가 반환한 에러 코드(예: 네트워크 오류, ID 오기입 등)를 분석합니다.

### 5-2. 라이선스 테스터 등록 (가상 결제 환경 세팅)
테스트 기기에서 실제 돈이 결제되는 대신 "항상 결제가 성공하는 가상 카드"를 사용할 수 있도록 개발자 계정을 설정해야 합니다.
- **라이선스 테스트 메뉴 이동:** Google Play 콘솔 메인 대시보드(전체 앱 목록 화면)의 왼쪽 메인 메뉴에서 가장 아래로 내려가 **설정 > 라이선스 테스트(License testing)**를 클릭합니다.
- **테스터 이메일 추가:** '테스터 추가' 영역에 테스트용 안드로이드 기기(스마트폰)에 로그인되어 있는 구글 계정 이메일 주소를 입력하고 엔터를 눌러 등록합니다.
- **응답 규칙 설정:** 라이선스 응답 규칙 항목이 `RESPOND_NORMALLY`(정상 응답)로 설정되어 있는지 확인합니다. 이 설정을 통해 구글 서버가 실제 출시 상태와 동일한 결제 프로세스를 시뮬레이션해 줍니다.

### 5-3. 가상 결제 진행 (샌드박스 테스트)
4단계에서 업로드했던 내부 테스트 트랙 링크를 통해 앱을 설치하고 결제 흐름을 밟아봅니다.
- **테스터 참여 링크 확인:** 플레이 콘솔 내 앱 대시보드의 **테스트 > 내부 테스트** 메뉴로 이동하여 '테스터' 탭에 있는 웹 버전 참여 링크(URL)를 복사합니다.
- **앱 설치:** 복사한 링크를 테스트 기기 브라우저에 붙여넣어 테스터 참여 수락 버튼을 누른 뒤, 연결되는 구글 플레이 스토어에서 앱을 다운로드합니다. (반드시 이 과정을 거쳐야 구글 서버가 테스터 앱으로 인식합니다.)
- **결제창 호출 및 승인:** 앱 내에서 광고 제거 버튼을 누릅니다. 이때 구글 플레이 결제 바텀시트가 올라오며 결제 수단에 **"테스트 카드(항상 승인됨)"** 또는 **"테스트 수단"**이라는 문구가 명시되어 있는지 반드시 눈으로 확인합니다.
- **결제 완료:** 결제 버튼을 누르면 실제 돈이 빠져나가지 않고 가상으로 결제가 승인되며, 앱 내부 코드가 정상 동작한다면 3일 이내 자동 환불을 방지하는 즉시 승인(Acknowledge) 절차가 백그라운드에서 완료됩니다.

### 5-4. 핵심 비즈니스 로직 최종 검증
광고 제거 상품의 목적에 맞게 UI와 데이터가 유기적으로 연동되는지 시나리오별로 테스트합니다.
- **즉시 소멸 검증:** 가상 결제가 완료되는 순간, 결제 완료 콜백을 받은 데이터 레이어의 상태값(`isAdFree`)이 즉시 변경되면서 화면에 노출 중이던 배너 광고 뷰가 즉시 숨김(**GONE**) 처리되는지 확인합니다. 화면 전환 시 나오던 전면 광고도 더 이상 트리거되지 않아야 합니다.
- **앱 재시작 및 로컬 저장소 검증:** 앱을 완전히 강제 종료했다가 다시 실행해 봅니다. 로컬 저장소(`SharedPreferences` 등)에 구매 상태가 안전하게 보존되어, 앱이 켜지자마자 광고가 없는 상태로 UI가 그려지는지 체크합니다.
- **재설치 및 구매 복원(Restore) 검증:** 기기에서 앱을 아예 삭제(데이터까지 초기화)한 후, 플레이 스토어 내부 테스트 트랙을 통해 다시 설치합니다. 앱이 처음 켜질 때 백그라운드에서 구글 서버에 과거 구매 내역을 쿼리(`queryPurchasesAsync`)하여, 사용자가 다시 결제 버튼을 누르지 않아도 자동으로 광고가 제거된 프리미엄 상태로 복원되는지 최종 확인합니다.

## 6. 최종 검토 및 출시 (Release & Launch)

### 6-1. 실제 애드몹 광고 단위 ID로 교체 (릴리스 환경 전환)
테스트 단계에서 사용했던 공용 테스트 ID를 제거하고, 1단계에서 발급받은 본인 계정 고유의 실제 광고 단위 ID를 소스 코드에 바인딩하는 과정입니다.
- **상수 파일 또는 환경 변수 확인:** 코드 내에 광고 단위 ID(Ad Unit ID)를 관리하는 전역 변수나 리소스 파일(`strings.xml` 등)을 찾습니다.
- **실제 ID 적용:** 범용 테스트 ID 대신, 1단계에서 구글 애드몹 콘솔을 통해 생성했던 실제 배너 광고 ID(`ca-app-pub-XXXXXXXXXXXXXXXX/XXXXXXXXXX`)와 전면 광고 ID를 복사하여 교체해 넣습니다.
- **앱 ID 교체 여부 재확인:** 2단계에서 `AndroidManifest.xml`에 입력했던 애드몹 앱 ID(`ca-app-pub-XXXXXXXXXXXXXXXX~XXXXXXXXXX`)가 실제 본인 계정의 값이 맞는지 한 번 더 대조합니다.

> [!CAUTION]
> **무효 트래픽 절대 주의 (가장 중요):** 실제 광고 ID로 교체한 순간부터는 개발자 본인의 기기나 에뮬레이터에서 앱을 실행해 광고를 띄우거나 터치하면 절대 안 됩니다. 구글 애드몹 시스템이 이를 부정행위로 감지하여 출시도 하기 전에 계정을 영구 정지시킬 수 있습니다. 실제 ID 상태에서의 UI 확인이 꼭 필요하다면, 애드몹 콘솔의 **'설정 > 테스트 기기'** 메뉴에 본인 스마트폰의 광고 ID(GAID)를 반드시 사전에 등록해야 합니다.

### 6-2. 프로덕션 트랙으로 앱 출시 승인 요청 (스토어 배포)
내부 테스트 환경에서 검증이 완료된 빌드를 실제 일반 대중에게 공개하기 위해 구글 플레이 팀의 최종 심사를 받는 단계입니다.
- **최종 릴리스 빌드 생성:** 안드로이드 스튜디오에서 실제 광고 ID가 반영된 상태로 버전 코드(`versionCode`)를 상향 조정한 뒤, 다시 한번 서명된 Android App Bundle(AAB) 파일을 생성합니다.
- **프로덕션 메뉴 진입:** Google Play 콘솔의 해당 앱 대시보드에서 왼쪽 메뉴의 **출시 > 프로덕션(Production)**을 클릭합니다.
- **새 버전 출시 생성:** 우측 상단의 **새 버전 출시 만들기** 버튼을 누르고 새로 생성한 프로덕션용 AAB 파일을 업로드합니다.
- **인앱 상품 판매 상태 점검:** 4단계에서 만들어 둔 광고 제거 상품 ID(`remove_ads`)가 여전히 '활성' 상태로 정상 유지되고 있는지 다시 한번 확인합니다. (앱이 프로덕션 심사를 통과하면 인앱 상품도 동시에 일반 유저에게 판매가 시작됩니다.)
- **스토어 등록정보 및 개인정보처리방침 작성:**
    - 앱의 설명, 스크린샷, 고해상도 아이콘을 등록합니다.
    - 인앱 결제가 포함된 앱은 이용자 보호를 위해 **개인정보처리방침(Privacy Policy)** URL 입력이 필수입니다.
- **심사 제출:** 하단의 **버전 검토**를 누른 뒤 **프로덕션 트랙으로 출시 시작**을 클릭합니다. 구글의 보안 및 정책 심사(보통 1일~7일 소요)가 완료되면 플레이 스토어에 앱과 광고 제거 결제 기능이 공식적으로 론칭됩니다.
